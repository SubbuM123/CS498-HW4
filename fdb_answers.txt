Q1: FDB decouples the TMS and SS, to enable independent scaling. FDB decouples these two so it is stateless and so that it can “discard the old TS and start a new one, avoiding expensive log replays entirely”. The TMS layer processes read and write transactions in memory, orders them, and detects conflict. The SS servers to store data and help with reads of the data.
Q2: The resolver is responsible for detecting read-write conflicts between concurrent transactions. The resolver checks this at commit time, after the proxy asks the sequencer for a commit version but before the commit is actually done to the database.
Q3: The purpose of the read version is to allow for MVCC and give a snapshot of the data. The point of the read version is for the resolver to check that no keys between the read and write versions were affected by any transactions. If they were, that means it is not safe to commit, and the transaction aborts.This guarantees FDB consistency because we don’t allow multiple transactions to modify the data into invalid states.
Q4: Snapshot reads are different from standard reads because they do not add to the transaction’s read conflict ranges, which means you avoid read-write aborts. You would choose snapshot reads if you want to optimize reading for background tasks, analytics, or when you do not need strict serializability. With a standard read, you cannot do these tasks as easily, since the transaction may be aborted by the read.
Q5: It is cheaper because the TS can determine the Recovery Version from the Log System, while the Storage Servers continue pulling data in the background. This means recovery becomes a cheap metadata operation, instead of a log replay. Overall, since the TS has no state, you can always discard the old TS, start a new one, and just do the metadata operation to recover. The SS always pulls data from the LS to find the best recovery version.
